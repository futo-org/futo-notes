## 1. Goal

FUTO Notes is going to **three runtimes**: native iOS (SwiftUI), native Android
(Compose, new), and Tauri desktop (Svelte). The objective is to make the **note
business logic a single source of truth in Rust**, consumed by all three, so that
UI code (Swift / Kotlin / TS-Svelte) is presentation only.

Today the rules (filename sanitization, tag parsing, title semantics, CRUD) exist in
**three** places — TS (`packages/shared`), Rust (`crates/futo-notes-core/src/files.rs`),
and the iOS spike's Swift (`apps/ios/Sources/NotesStore.swift`). Adding
Kotlin would make four. This drift surface is exactly what `AGENTS.md` is paranoid
about ("the filename IS the title"). The fix is to collapse the rules + CRUD into Rust
and bind them three ways.

### What stays where

- **Rust** owns: filename/tag/title rules, note + folder CRUD, file I/O, crypto, sync
  protocol + orchestration, full-text + vector search, hashing, merge.
- **Native shells (Swift/Kotlin)** own: reactive UI state, debounce/animation,
  platform quirks (e.g. the iOS keyboard swizzle), and hosting the web editor.
- **Tauri TS frontend** owns: reactive `$state` caches, debounce/polling, watcher
  coordination — calling Rust through `#[tauri::command]` (IPC round-trip on desktop
  is explicitly acceptable; confirmed decision).
- **The editor** (CodeMirror 6 / Svelte) stays web, bundled to a single `editor.html`,
  hosted in a `WKWebView`/`WebView` on native and embedded in the Tauri frontend.

## 2. Settled decisions

From the design discussion + an explicit decision round:

| # | Decision | Choice |
|---|---|---|
| D1 | Repo topology | **Single monorepo.** Do NOT split into per-component repos. Boundaries fall where release cadence/ownership diverge, not where languages differ; the components here change together (UniFFI surface, editor bridge, Rust core). The server already being a separate repo is the one correct split. |
| D2 | Tauri ↔ Rust | Tauri's Rust backend consumes the same Rust core via `#[tauri::command]`. IPC round-trip on desktop is fine. |
| D3 | Browser target | **None.** No browser-only deployment → **no WASM build needed.** (The browser path is dev/test only — Playwright, factory-judge.) |
| D4 | Native search | **Rust `index` crate** (tantivy or simple inverted index). One implementation shared by desktop + both native apps. |
| D5 | Near-term order | **Consolidate core first** — finish Phases 0–3 before starting Android (Phase 4), so Android builds on a proven single-source core. |
| D6 | Cutover safety | **Feature-gate then flip.** Keep both old (TS/Swift) and new (Rust) impls behind a `rust_model` flag, diff-test them, flip in a single commit per platform. |
| D7 | Rules crate home | **New `futo-notes-model` crate.** `core` stays low-level primitives; `model` owns the note domain. Reuse `core::files`/`hash`/`invariants` rather than re-port. |
| D8 | Type generation | Add **tauri-specta** to the Tauri backend to generate TS types from `#[tauri::command]` signatures. UniFFI generates Swift/Kotlin for native. |

## 3. Target repo shape

```
futo-notes/                      # one monorepo
├── crates/                      # pure Rust workspace
│   ├── futo-notes-core/         # primitives: e2ee, hash, files, merge, sync-wire types, invariants
│   ├── futo-notes-model/        # NEW: note domain — filename/tag rules, title, note+folder CRUD, wikilinks
│   ├── futo-notes-index/        # NEW: full-text + preview index, search chunking/RRF, graph layout
│   ├── futo-notes-sync/         # SHRINKS to HTTP client + wire protocol only (orchestration moves out)
│   ├── futo-notes-inference/    # ONNX embeddings (existing)
│   ├── futo-notes-app/          # NEW: composition root — wires model+index+sync+inference, owns orchestration
│   └── futo-notes-ffi/          # NEW: thin UniFFI facade over -app, for iOS + Android
├── packages/
│   └── editor/                  # NEW (formalize existing build): Svelte/CM6 editor + futoBridge contract
│                                #   (packages/shared rules DELETED once Phase 3 lands; auth types → tauri-specta)
├── apps/
│   ├── tauri/                   # desktop: Svelte frontend (chrome + editor) + Rust backend (deps -app)
│   ├── ios/                     # SwiftUI chrome + editor WebView + ffi.xcframework (promote native-ios-spike)
│   └── android/                 # NEW: Compose chrome + editor WebView + ffi .so + Kotlin bindings
```

### Rust dependency graph (no cycles)

```
core ──┬─ model  → core                  (filename/tag rules, note+folder CRUD, wikilinks)
       ├─ index  → core, inference        (full-text + vector search, graph)
       ├─ sync   → core                   (HTTP client + wire types ONLY)
       └─ inference
              app → model, index, sync     (orchestration: pull→merge→write→push→checkpoint)
              ffi → app                     (thin UniFFI for iOS + Android)
   apps/tauri/src-tauri → app  (+ tauri-specta)   (thin #[tauri::command])
```

Key move: the sync **orchestrator** (`futo-notes-sync/src/orchestrator.rs`) relocates
to `app`, and `state.rs` (object-map persistence) relocates to `model`. That leaves
`sync` as a dependency-free HTTP/protocol client and breaks the would-be cycle.

## 4. ⚠️ Pull forward NOW — independent data-loss hotfix

`crates/futo-notes-sync/src/orchestrator.rs:14-17` — on a **dirty 3-way merge the
spike keeps the remote version and silently discards the user's local edits.** This is
live data loss in the current iOS spike. Fix independently of the refactor:

- Before applying remote on a dirty merge, write a `note (conflict YYYY-MM-DD).md`
  copy of the local content via `core::files` atomic write (this matches the full
  Tauri client's behavior).
- Add a regression test that reproduces the loss before the fix.
- ~Half a day, solo. Do not wait for Phase 5.

## 5. Phased plan

> Cutover uses a `rust_model` feature gate (D6) through Phases 2–3: old and new impls
> coexist and are diff-tested, then flipped in one commit per platform.

### Phase 0 — Foundation: conformance harness, crate skeletons, type generation
**Goal:** make rule drift impossible to merge before any logic moves.

- Extract the 66 existing TS test cases into language-neutral JSON golden fixtures:
  - `packages/shared/src/filename.test.ts` (35 tests)
  - `packages/shared/src/tags.test.ts` (24 tests)
  - `packages/shared/src/image.test.ts` (7 tests)
  - → `tests/conformance/{filename,tags,image}.json`, read by **both** Vitest and a
    Rust `tests/*_conformance.rs`. TS passes immediately; Rust stubs fail (red baseline).
- Stub the four new crates (`model`, `index`, `app`, `ffi`) compiling-but-empty in the
  Cargo workspace.
- Add **tauri-specta** to `apps/tauri/src-tauri` + `build.rs`, generating `bindings.ts`
  from the existing 14 commands (`apps/tauri/src-tauri/src/core.rs:304-1298`, currently
  untyped on the frontend). Independent of the rest; de-risks the desktop type story.
- **Editor track (parallel):** formalize the working `vite.editor.config.ts → editor.html`
  pipeline into `packages/editor`; promote the `futoBridge` postMessage contract +
  `window.FutoEditor` API (`src/editor-embed/main.ts:88-112`) into a **versioned
  interface file**. Three hosts depend on it.

**Workflow:** mostly solo (scaffolding). Small fan-out only for fixture extraction
(one agent per `*.test.ts` → JSON).
**Verify:** `just test-shared`, `cargo build`, `just build`.

### Phase 1 — Port deterministic rules to Rust `futo-notes-model`
**Goal:** one canonical rules implementation, proven bit-for-bit against TS.

- Port into `futo-notes-model`, **reconciling with existing `core/files.rs`** (1150 LOC
  already has `sanitize`, `validate`, `ensure_safe_note_id`, unique-id collision —
  verify these match the TS rules; do not blind re-port):
  - filename: `sanitizeTitle` (`filename.ts:71`), `validateTitle` (`:81`),
    `isWindowsReservedName` (`:56`), `validateFolderName` (`:134`),
    `hasCaseInsensitiveSiblingCollision` (`:157`), `validateFolderPath` (`:172`),
    `pathDepth` (`:209`); constants `MAX_TITLE_LENGTH=200`, `MAX_FOLDER_DEPTH=10`,
    `FALLBACK_TITLE='Untitled'`, `FORBIDDEN_CHARS_RE`.
  - tags: `extractTags` (`tags.ts:111`), `extractHeaderTagBlock` (`:139`),
    `normalizeTagName` (`:35`), `isValidTagName` (`:23`); `TAG_REGEX`, `MAX_TAG_LENGTH=50`.
  - image: `isImageFilename` (`sync.ts:6`), `IMAGE_EXTENSIONS`.
- **Landmine 1 — `TAG_REGEX` (`tags.ts:18`) uses lookahead.** Rust's `regex` crate does
  not support lookaround. Use `fancy-regex` (already a dependency via the inference
  tokenizer) or a hand-rolled scan.
- **Landmine 2 — JS UTF-16 vs Rust UTF-8 offsets.** `extractHeaderTagBlock` returns a
  byte offset used for file I/O; compute in bytes in Rust. `.replace()`/`.length` in JS
  are UTF-16-based.
- **Landmine 3 — tag canonicalization** lowercases (`#Demo`→`#demo`); must dedupe
  identically or sync state diverges.

**Workflow: YES — highest-value workflow in the plan.** Three independent rule modules →
`pipeline([filename, tags, image], port→Rust, adversarially-verify)`. The verify stage
is perspective-diverse: run the golden fixtures **plus** fuzz inputs (emoji, control
chars, nested/unclosed code fences, case-collision, `../../etc` traversal) and diff Rust
output vs the TS reference. **Save this run as `/conformance-check`** in
`.claude/workflows/` so CI/team re-run it on every rule change.
**Verify:** `just test-rust`, `just test-shared`.

### Phase 2 — `model` CRUD + rewrite iOS `NotesStore` as a thin wrapper
**Goal:** Swift stops doing business logic; becomes a reactive shell like `SyncManager`.

- Build CRUD in `model` (reuse `core::files`, `core::invariants`; move `state.rs` here):
  `create/read/write/delete/rename/move`, `scan → Vec<NoteMetadata>`, preview extraction,
  unique-id collision, folder ops, wikilink parsing. Define
  `struct NoteMetadata { id, title, folder, modified, preview, tags }`.
- Compose in `app`; expose through `ffi` (UniFFI) alongside the existing `SyncClient`.
- Rewrite `apps/ios/Sources/NotesStore.swift` (439 lines) call-by-call to
  FFI: `sanitizeTitle` (`:32`), `extractTags` (`:57`), `makeId/split` (`:90`),
  `uniqueId` (`:117`), `scanFiles/reload` (`:194`–`254`), `read/write` (`:303`–`347`),
  `createNote/delete/rename/moveNote` (`:350`–`421`).
- **Keep in Swift** (genuine UI concerns): `@Published` state, the 400 ms save debounce
  (`NoteEditorView.swift:100`), the in-place row-update optimization (`NotesStore.swift:333`),
  and the keyboard swizzle (`EditorWebView.swift:257`).

**Workflow: YES — two-stage, gated.** (1) Build `model`/`app`/`ffi` exports (parallel by
concern). (2) After Rust compiles, a behavioral-parity workflow turning each enumerated
risk into a test, one verifier per behavior comparing new-Rust vs old-Swift output.
**Behavioral-parity risk checklist** (must match old Swift exactly): note list sort
(mtime desc), `makePreview` whitespace collapse, tag lowercasing/dedup, folder-ancestor
surfacing (`registerAncestors`), empty-folder surfacing (`scanDirs`), path traversal
rejection, unique-id collision strategy (`Foo`→`Foo-2`→`Foo-3`), UTF-8 emoji round-trip.
**Verify:** `just test-mac-ios`, `just test-rust`.

### Phase 3 — Route Tauri's note model through Rust (desktop parity)
**Goal:** delete the second copy of the rules; desktop consumes the same `app` crate.

- Add `#[tauri::command]` wrappers over `model` rules + CRUD; tauri-specta regenerates
  `bindings.ts`.
- Replace every `@futo-notes/shared` rule import in `src/lib` with the generated command
  calls — behind the existing `src/lib/fileSystem.ts` shim plus a new `rules.ts` shim, so
  `notes.svelte.ts`, `notesIndex.ts`, `folders.svelte.ts`, and the reactive layer **do
  not change**. Then delete `packages/shared/{filename,tags}.ts`.
- Reactive orchestration stays TS: `notes.svelte.ts`, `notesIndex.ts`, `searchIndex.ts`
  (until index moves — see below), `autoSyncV2.ts`, `syncCoordinator.ts`,
  `syncManager.svelte.ts`, `folders.svelte.ts`, `wikilinks.ts`.
- **Search note (D4):** the Rust `index` crate becomes the search backend on native;
  on Tauri, decide during this phase whether to route search through Rust commands too
  or keep MiniSearch (`searchIndex.ts`) for desktop. Default: build the Rust `index`
  crate, expose via command, retire MiniSearch once parity is proven.

**Workflow: YES — migration fan-out.** Scout call sites inline, then
`pipeline(callSites, migrate→command-shim, verify build+test per file)`. **Adversarial
check:** flag any call site that would now do **per-keystroke IPC** (e.g. `extractTags`
on every change) — those should stay TS or move into the editor, per the
TS-first/avoid-the-round-trip principle.
**Verify:** `just test-e2e`, `just test-cross-platform`.

### Phase 4 — Native Android shell (after 0–3, per D5)
**Goal:** second native platform on the same Rust core + same editor bundle.

- `scripts/build-rust-android.sh`: `cargo ndk --platform 24 --target
  aarch64-linux-android,armv7-linux-android,x86_64-linux-android build -p futo-notes-ffi`
  → `uniffi-bindgen --language kotlin` → copy `.so` into `jniLibs/<abi>/`. The per-ABI

- `apps/android`: Compose chrome + `WebView` hosting the **same** `editor.html` (single

  contract.
- `futo-notes-ffi/Cargo.toml` needs `crate-type = [staticlib, cdylib, lib]` (the spike's
  `futo-notes-sync` already does — copy the pattern).
- ⚠️ Requires `ANDROID_NDK_HOME`; the build script must check it early with a clear error.

**Workflow: MIXED.** Rust/NDK/Gradle plumbing is sequential/env-dependent → solo. The
Compose UI port mirrors the three SwiftUI screens (`NoteListView`, `NoteEditorView`,
`SyncView`) → a workflow with one agent per screen (Swift version as spec) is a clean
parallel port.
**Verify:** `just test-cross-platform-android` + device QA.

### Phase 5 — Finish sync hardening in `app`

## 6. Dynamic workflows — how & when

Workflows run in the **background** (session stays responsive), keep intermediate
results in script variables (not the main context), are **resumable within a session**
(completed agents return cached results), and can be **saved as `/<name>` commands**
(`.claude/workflows/` for the repo). Hard constraint: **no mid-run user input** — so each
phase that needs sign-off is its **own** workflow with a review gate between, never one
mega-run.

| Phase | Workflow | Pattern | Rationale |
|---|---|---|---|
| 0 | minimal | small fan-out (fixture extraction) | mostly mechanical scaffolding |
| 1 | **yes** | pipeline + adversarial/fuzz verify | rule drift silently corrupts sync |
| 2 | **yes** (×2, gated) | build fan-out, then behavioral-parity verify | 20+ call sites + enumerated risks |
| 3 | **yes** | migration fan-out + IPC-hotpath check | many call sites across `src/lib` |
| 4 | partial | parallel UI port (Compose screens) | Rust/Gradle plumbing is sequential |
| 5 | **yes** | adversarial scenario sweep | sync correctness, data-loss-adjacent |

**Reusable commands to save:** `/conformance-check` (Phase 1), `/sync-adversarial`
(Phase 5), and the `migration-survey` script (already saved — see §8).

## 9. Deferred / still-open

- **Editor as `packages/editor`:** the single-file build (`vite.editor.config.ts`)
  already works for iOS; formalizing it as a workspace package + versioned bridge is
  Phase 0 but low-risk to defer to when Android lands.

  `.app-state.json` for auto-sync re-derivation (sandboxed, low risk on mobile). iOS
  Keychain persistence is being handled separately (another agent). Android will need
  Keystore. Consider making credentials Rust-owned long-term (out of scope here).
- **Crate granularity:** `index` may start folded inside `model` and split out when
  search grows; `app` may start thin. Don't pre-fragment.
