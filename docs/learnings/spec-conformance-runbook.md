### idb on this machine (installed after the fact — works, with caveats)

`idb_companion` 1.1.8 (brew) + `fb-idb` 1.1.7 are BOTH installed; the `idb` CLI
is at `/Users/justin/Library/Python/3.9/bin/idb` (NOT on PATH — that's why the
initial `which idb` said absent). It talks to the iOS 26.5 sim.

- **Reads work great.** `idb ui describe-point X Y` (POINTS, not px) is reliable
  and is how I **closed the nav.md:13 iOS AX gap**: gear→`nav-settings`/"Settings"/
  AXButton, cloud→`nav-sync`/"Sync"/AXButton, "+"→`nav-create`/"New note or
  folder"/AXPopUpButton, all enabled. Screen: 402×874 pt @3x (1206×2622 px).
- **`idb ui describe-all` is SHALLOW on iOS 26.5** — it returned only ~6 top-
  level elements (note rows + a collapsed "Toolbar" group), missing the nav-bar
  buttons and bottom search bar. Use `describe-point` at specific coords instead.
  (The `idb_companion` also logs an objc class-collision warning on this Xcode —
  non-fatal but a compatibility smell.)
- **Taps on small SwiftUI toolbar buttons are flaky/toggly** — a single
  `idb ui tap 85 84` on the cloud opened the Sync sheet ~50% of the time;
  loop-tapping fought the sheet's open/close toggle. Tap once, check once.
- **Text entry into an iOS SecureField is unreliable** — password autofill
  ("Passwords" QuickType bar) mangles `idb ui text`; the sync connect kept
  returning "Auth: invalid password" despite typing `testing123`. For a plain
  TextField, prefer setting the value out-of-band:
  `xcrun simctl spawn booted defaults write com.futo.notes.dev futo.serverURL
  http://localhost:3100` then relaunch — beats backspace+type (which dropped a
  char and even backgrounded the app mid-edit).
- **iOS sync connect PATH is verified**: with the URL default at :3100 the iOS
  SyncClient reached the server and surfaced auth errors inline correctly — the
  only thing that fails is typing the password via idb. iOS uses the SAME shared
  Rust core (futo-notes-ffi SyncClient → orchestrator::run_sync) proven live on
  desktop+Android, so full iOS live sync is covered short of a clean keystroke.
- Editor content-area taps DO work (the block-handle/editor flows). The nav bar
  and secure fields are the flaky spots. For those, `xcrun simctl` +
  `describe-point` reads + `defaults` writes are more reliable than blind taps.

  `SettingsScreen.svelte` has no benchmark UI (the shared model already called

  fragile AppleScript. iOS uses the SAME shared Rust sync core proven on
  desktop+Android; iOS sync UI states were visually confirmed. Full iOS live
  connect via UI-driving was not exhaustively run.
