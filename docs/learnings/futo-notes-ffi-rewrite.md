# FUTO Notes FFI Rewrite Lessons

## Why the rewrite was needed

`futo-notes-ffi` is the UniFFI boundary shared by the native SwiftUI and Compose
shells. Its original crate root combined every exported record, conversion,
rule, note-store method, sync callback, and sync-client method in one file.
Although the implementation was mostly thin, the layout obscured ownership and
made unrelated changes converge on the same architectural center.

The rewrite preserved the external UniFFI contract while organizing the crate
around the two capabilities it projects:

```text
src/
  lib.rs
  notes/
    mod.rs
    contract.rs
    rules.rs
    store.rs
  sync/
    mod.rs
    contract.rs
    events.rs
    client.rs
  bin/
    uniffi-bindgen.rs
tests/
  note_contract.rs
  sync_contract.rs
  support/
    mod.rs
```

The crate root is now only the public facade and UniFFI scaffolding entry. The
`notes` and `sync` modules each own one coherent projection.

## Ownership and dependency direction

The FFI crate does not own note or sync domain behavior:

- `futo-notes-model` owns deterministic note rules.
- `futo-notes-store` owns the vault, note workflows, and search lifecycle.
- `futo-notes-sync` owns synchronization and live-sync behavior.
- `futo-notes-ffi` translates those owners into UniFFI-safe objects, records,
  errors, callbacks, and results.

`NoteStore` wraps one `LocalNoteStore`; it does not recreate note workflows or
maintain a second search owner. `SyncClient` wraps one `SyncSession`; it does
not mirror sync state or protocol decisions.

`futo-notes-core` remains part of the compiled native libraries through the
model, store, and sync crates. It is intentionally not a direct FFI dependency
because the FFI crate has no direct core imports. Keeping only the actual
dependency edges prevents the boundary from bypassing the owning crates.

## Preserve semantic compatibility, not generated layout

The product contract is the complete UniFFI semantic surface:

- Exported objects and free functions.
- Method names, parameters, and sync/async shapes.
- Records and every field.
- Enums and errors with every variant.
- Callback methods and ownership.
- Result and failure shapes.
- Threading, lifetime, and panic-boundary behavior.

Current Swift and Kotlin call sites are only a lower bound on that contract.
An exported capability must not be removed merely because a current shell does
not call or render it.

Generated Swift, Kotlin, C headers, checksums, JNI libraries, and XCFrameworks
are build artifacts rather than hand-maintained compatibility surfaces.
Documentation or source-layout changes can alter generated checksums without
changing the semantic API. Regenerate these artifacts from the preserved Rust
surface and verify the native consumers instead of freezing generated bytes or
editing generated output.

## Keep the boundary thin without creating a facade warehouse

Thin does not mean that every projection belongs in one file. A useful split is:

- `contract.rs`: exported records, errors, enums, and exhaustive conversions.
- `rules.rs`: stateless delegation to canonical note rules.
- `store.rs`: the stateful note/store/search object and its orchestration.
- `events.rs`: the callback contract and lower-layer listener bridge.
- `client.rs`: the stateful sync object and its lifecycle orchestration.
- `mod.rs`: private capability facade with deliberate re-exports.
- `lib.rs`: public crate facade and one `setup_scaffolding!` invocation.

This keeps state and lifecycle beside the object that owns them while keeping
pure translations separate from effects. It also prevents the crate root from
becoming another implementation warehouse.

## Test the public boundary with non-vacuous promises

The FFI tests exercise the public Rust surface rather than private module
seams. They guard:

- Delegation of deterministic note rules to the canonical model.
- Complete note workflow results and safety outcomes.
- Record, enum, error, callback, and result shapes.
- `Send + Sync` expectations for exported stateful objects.
- Stable disconnected sync-client lifecycle behavior.
- Existing note content becoming searchable through BM25 after bootstrap.

Asynchronous search requires particular care. An assertion such as
`hits.iter().all(|hit| hit.source == "bm25")` passes for an empty result list
and therefore proves nothing. The public search contract test:

1. Creates a note containing a unique body term before bootstrap.
2. Bootstraps the store.
3. Waits with a bounded timeout for keyword readiness.
4. Searches for the unique term.
5. Requires the expected note with `source == "bm25"`.

Readiness polling should use a short sleep and a failure deadline. The deadline
is a bound, not a fixed delay; the test continues immediately once the index is
ready.

Keep distinct behaviors in distinct tests. Initial bootstrap indexing,
mutation-to-index propagation, and explicit rescan behavior have different
causes and should not be conflated in one broad workflow test.

## Native verification is part of the FFI contract

Rust tests alone cannot prove that UniFFI generation and native consumers still
work. FFI-visible changes should verify both binding paths:

```sh
just build-rust-ios
just build-ios-native
just build-rust-android
just build-android-native
just test-android-native
```

The build profiles are safety constraints:

- iOS uses the development profile.
- Android uses `release-ffi`.
- Do not switch FFI builds to the workspace's ordinary release profile because
  its aborting panic behavior is incompatible with UniFFI's unwind boundary.

Generated binding directories and native libraries remain ignored and must not
be committed.

## Source comments should carry boundary constraints

The final source-comment census retained only information that names and types
do not express:

- Stable wire identifiers and caller obligations for generated bindings.
- Worker-thread callback and nonblocking lifecycle constraints.
- Atomic-write, no-replace, push-first, and disconnect-reconciliation safety.
- Native build-profile, deployment-target, toolchain, and platform constraints.

Constructor narration, module summaries already expressed by the source tree,
and repeated descriptions of build commands were removed.

## Practical checklist for future FFI changes

- Put behavior in model, store, or sync; keep FFI as projection and lifecycle
  orchestration.
- Preserve the complete exported semantic surface unless a product/spec change
  explicitly authorizes otherwise.
- Update exhaustive conversions whenever lower-layer records or enums change.
- Add or update a public contract test for each changed semantic promise.
- Make asynchronous tests require an observable result, not only readiness or a
  vacuously true collection assertion.
- Regenerate Swift and Kotlin bindings after every FFI-visible change.
- Compile both native consumers and run Android JVM acceptance tests.
- Preserve the unwind-compatible iOS and Android build profiles.
- Never edit or commit generated bindings, JNI libraries, or XCFrameworks.
